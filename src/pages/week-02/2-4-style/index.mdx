## üéØ Writing Code for Humans

Here's a truth that surprises new developers: **you spend more time reading code than writing it**. In professional environments, you'll read your teammates' code, debug legacy systems, and revisit your own work months later.

Code that only the computer understands is a liability. Code that *humans* understand is an asset.

<div class="callout-insight">
  <span class="callout-icon">üí°</span>
  <div>
    <strong>The Real Audience</strong>
    <p>The compiler doesn't care about style. It processes <code>int x=1+2;</code> the same as properly formatted code. But the next developer who reads it‚Äîoften future you‚Äîabsolutely cares.</p>
  </div>
</div>

## üìù The Three Levels of C# Comments

C# provides three distinct commenting mechanisms, each with a specific purpose:

<div class="comment-levels">
  <div class="comment-level">
    <div class="level-header">
      <span class="level-icon">1Ô∏è‚É£</span>
      <h3>Single-Line Comments <code>//</code></h3>
    </div>
    <div class="level-content">
      <p><strong>Purpose:</strong> Quick notes on specific logic lines</p>
      <p><strong>Use when:</strong> Explaining <em>why</em> a particular approach was chosen</p>
      
```csharp
// Calculate discount for loyalty members (business rule #247)
double discount = basePrice * 0.15;

int retryCount = 3; // Max retries before timeout (per SLA)
```
      
      <div class="level-tip">
        <strong>Pro Tip:</strong> Place on their own line for complex explanations, or at end of line for brief clarifications.
      </div>
    </div>
  </div>
  
  <div class="comment-level">
    <div class="level-header">
      <span class="level-icon">2Ô∏è‚É£</span>
      <h3>Multi-Line Comments <code>/* */</code></h3>
    </div>
    <div class="level-content">
      <p><strong>Purpose:</strong> Longer explanations or temporarily disabling code blocks</p>
      <p><strong>Use when:</strong> You need multiple lines, or you're debugging by commenting out sections</p>
      
```csharp
/*
 * Legacy pricing algorithm - DO NOT REMOVE
 * This handles edge cases from the 2019 migration.
 * Scheduled for refactoring in Q3 2026.
 * Contact: architecture@company.com
 */

/* Temporarily disabled for testing
double oldPrice = CalculateOldPrice(item);
ApplyOldDiscount(ref oldPrice);
*/
```
      
      <div class="level-warning">
        <strong>Caution:</strong> Multi-line comments cannot be nested. <code>/* outer /* inner */ */</code> will cause errors.
      </div>
    </div>
  </div>
  
  <div class="comment-level highlight">
    <div class="level-header">
      <span class="level-icon">3Ô∏è‚É£</span>
      <h3>XML Documentation <code>///</code></h3>
    </div>
    <div class="level-content">
      <p><strong>Purpose:</strong> Formal documentation for methods, classes, and properties</p>
      <p><strong>Use when:</strong> Defining public APIs, library code, or anything other developers will consume</p>
      
```csharp
/// <summary>
/// Calculates the final price after applying all discounts.
/// </summary>
/// <param name="basePrice">The original item price before discounts.</param>
/// <param name="customerTier">The customer's loyalty tier (1-3).</param>
/// <returns>The final price after tier-based discount is applied.</returns>
/// <exception cref="ArgumentException">Thrown when tier is outside valid range.</exception>
public double CalculateFinalPrice(double basePrice, int customerTier)
{
    // Implementation here
}
```
      
      <div class="level-special">
        <strong>The Magic:</strong> XML comments are parsed by the compiler to generate:
        <ul>
          <li>IntelliSense tooltips in Visual Studio / VS Code</li>
          <li>Automated API documentation (like Microsoft's docs)</li>
          <li>Technical reference manuals</li>
        </ul>
      </div>
    </div>
  </div>
</div>

## üèõÔ∏è The Architect's Perspective: Self-Documenting Code

<div class="architect-note">
  <div class="architect-avatar">üë∑‚Äç‚ôÇÔ∏è</div>
  <div class="architect-content">
    <h4>Comments Explain "Why," Not "What"</h4>
    <p>
      Here's a principle that separates junior developers from senior engineers: <strong>the code itself should explain what it does</strong>. Comments should explain <em>why</em>.
    </p>
    
    <div class="bad-example">
      <span class="example-label">‚ùå Bad: Restating the Code</span>
```csharp
// Add 1 to counter
counter = counter + 1;

// Set name to "John"
string name = "John";
```
      <p>These comments add no value‚Äîthey just repeat what's obvious from reading the code.</p>
    </div>
    
    <div class="good-example">
      <span class="example-label">‚úÖ Good: Explaining Intent</span>
```csharp
// Increment to account for zero-based indexing in display
counter = counter + 1;

// Default name for anonymous checkout (GDPR compliance)
string name = "John";
```
      <p>Now the reader understands the business logic behind the code.</p>
    </div>
  </div>
</div>

## üé® Code Formatting Conventions

Beyond comments, professional C# follows strict formatting rules:

### Naming Conventions

| Element | Convention | Example |
|---------|------------|---------|
| **Classes** | PascalCase | `CustomerAccount`, `OrderProcessor` |
| **Methods** | PascalCase | `CalculateTotal()`, `ProcessPayment()` |
| **Variables** | camelCase | `customerName`, `orderCount` |
| **Constants** | PascalCase or UPPER_CASE | `MaxRetries`, `DEFAULT_TIMEOUT` |
| **Private fields** | _camelCase | `_connectionString`, `_logger` |
| **Parameters** | camelCase | `void Process(int orderId)` |

### Bracing Style

C# uses the **Allman style** (braces on their own lines):

```csharp
// ‚úÖ Correct: Allman style (C# standard)
if (isValid)
{
    ProcessOrder();
}

// ‚ùå Avoid: K&R style (common in JavaScript, not C#)
if (isValid) {
    ProcessOrder();
}
```

### Spacing and Indentation

```csharp
// ‚úÖ Good: Clear spacing
public void ProcessOrder(int orderId, string customerName)
{
    if (orderId > 0)
    {
        var order = GetOrder(orderId);
        order.Process();
    }
}

// ‚ùå Bad: Cramped, hard to read
public void ProcessOrder(int orderId,string customerName){
if(orderId>0){var order=GetOrder(orderId);order.Process();}}
```

## üìã Common XML Documentation Tags

| Tag | Purpose | Example |
|-----|---------|---------|
| `<summary>` | Brief description | What the method/class does |
| `<param>` | Parameter description | `<param name="id">The user ID</param>` |
| `<returns>` | Return value description | What the method returns |
| `<exception>` | Possible exceptions | `<exception cref="ArgumentException">` |
| `<remarks>` | Additional details | Extended explanation |
| `<example>` | Usage examples | Code samples |
| `<see>` | Cross-reference | `<see cref="OtherMethod"/>` |
